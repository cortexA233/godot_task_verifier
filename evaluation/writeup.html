RoboBlast Grenade Verifier

A behavioral Godot 4.6 grader for a missing grenade weapon, with rollout evidence from Claude Code Sonnet 5 medium, Claude Code Opus 4.8 max, and Codex GPT-5.5 xhigh attempts, plus eight representative anti-cheat fake solutions, including a high-scoring visual-floor failure.

Task Chosen

The task removes the grenade weapon from RoboBlast and asks the rollout agent to rebuild it from a behavioral-only prompt. The expected player-facing behavior spans weapon switching, HUD state, trajectory preview, physical projectile flight, localized explosion damage, visual/audio feedback, cleanup, and preservation of default shooting/melee.

The feature is a good benchmark slice because it crosses code, scenes, resources, UI, physics, damage conventions, and visual presentation. It is also hard to grade honestly: a solution can look plausible while skipping aim reactivity, spatial damage, cooldown behavior, or main-scene integration.

The agent-facing prompt is TASK_PROMPT.md. It intentionally describes outcomes only: no historical file names, node paths, class names, methods, or verifier internals.

Verifier Design

The verifier is external to the candidate project. It copies the candidate into a temporary directory, injects verifier_godot/__verifier__, runs Godot headlessly, and writes a structured 0-100 JSON result.

Scoring is category-based so near-misses receive useful partial credit instead of a single pass/fail result. The current rubric is:

Rollout Runs

Rollout workspaces were prepared as isolated copies with verifier files, local git history, generated artifacts, and assignment-only materials stripped from the agent-facing workspace. The local Godot MCP configuration for the task project enables @coding-solo/godot-mcp through Node and points GODOT_PATH at the Godot 4.6 installation used for the run. Codex runs had that same Godot MCP configuration available, even when the run record's final evidence came from shell/static checks rather than direct editor-driving MCP calls.

The rollout setup and evidence capture used the verifier repository's two repo-local skills. Before each evaluated run, skills/prepare-agent-run-workspace/ created an isolated workspace/ plus evaluator-owned evidence/ directory from the ablated task. After the agent stopped, skills/collect-agent-run-evidence/ captured the objective run record: git status, binary-capable diff, final SHA, manifest updates, score JSON/PDF/log paths, and the exact grader command. Future evaluated agent runs should use this same two-skill workflow; ad hoc manual copies should be treated as an exception and documented in the run notes.

The retained evidence set contains nine agent-run branches: three Claude Code Sonnet 5 medium runs, three Claude Code Opus 4.8 max runs, and three Codex GPT-5 xhigh runs. Reproduction Commands

Verifier repository: use the operator's live verifier checkout for new runs. Future generated writeups should retain this verifier-repository entry, the isolated-run skill workflow, the complete grader command pattern below, and the rule that docs must use placeholders instead of machine-specific local absolute paths.
  1. Set the verifier root, Godot 4.6 console executable, ablated project path, task prompt path, and a fresh run root.
  2. Prepare the isolated agent workspace with prepare-agent-run-workspace. Give the rollout agent only $RunRoot\workspace and the prompt text copied into $RunRoot\evidence\prompt-for-agent.md.
  3. Keep $Verifier and $RunRoot\evidence operator-only while the agent is running.
  4. After the agent stops, run collect-agent-run-evidence once to capture git status, diff, final SHA, manifest, and transcript-facing artifacts.
  5. Run run_grader.py against the completed workspace, then run collect-agent-run-evidence again with the score JSON and exact grader command.
  6. Review score.json, score-report.pdf, godot-verifier.log, diff.patch, run-manifest.json, and any screenshot-probe outputs under $RunRoot\evidence.
$Verifier = "<verifier-repo>" $Godot = "<godot-4.6-console-executable>" $AblatedProject = "<ablated-task-project>" $TaskPrompt = "$AblatedProject\TASK_PROMPT.md" $RunRoot = "<agent-runs-root>\run-01-cc-sonnet" python "$Verifier\skills\prepare-agent-run-workspace\scripts\setup_agent_run.py" ` --source "$AblatedProject" ` --run-root "$RunRoot" ` --agent cc-sonnet ` --model "model/version if known" ` --tool "Godot MCP available" ` --godot-mcp available ` --prompt "$TaskPrompt" python "$Verifier\skills\collect-agent-run-evidence\scripts\finalize_agent_run.py" ` --run-root "$RunRoot" $ScoreJson = "$RunRoot\evidence\score.json" $PdfReport = "$RunRoot\evidence\score-report.pdf" $Log = "$RunRoot\evidence\godot-verifier.log" python "$Verifier\run_grader.py" ` --project "$RunRoot\workspace" ` --godot "$Godot" ` --verifier-root "$Verifier" ` --out "$ScoreJson" ` --pdf-report "$PdfReport" ` --log "$Log" $GraderCommand = @" python "$Verifier\run_grader.py" --project "$RunRoot\workspace" --godot "$Godot" --verifier-root "$Verifier" --out "$ScoreJson" --pdf-report "$PdfReport" --log "$Log" "@ python "$Verifier\skills\collect-agent-run-evidence\scripts\finalize_agent_run.py" ` --run-root "$RunRoot" ` --score-json "$ScoreJson" ` --grader-command "$GraderCommand"

For reference, ablated-task, anti-cheat probe, or retained-run audits, use the same grader command pattern with the candidate project path and output names under evaluation/evidence/:

$Verifier = "<verifier-repo>" $Godot = "<path-to-godot-4.6-console-executable>" $Project = "<path-to-candidate-or-probe-project>" $OutJson = "$Verifier\artifacts\calibration-20260703-name-score.json" $OutLog = "$Verifier\artifacts\calibration-20260703-name-score.log" python "$Verifier\run_grader.py" --project "$Project" --godot "$Godot" --verifier-root "$Verifier" --out "$OutJson" --log "$OutLog"

python "$Verifier\export_debug_arena.py" --project "$Project" --out "$Verifier\artifacts\debug-arena"
