Take-Home Assignment

Build an agent-coding evaluation task for a real game

Take a real, open-source 3D game. Turn one of its elements — code, animation, a 3D model, UI, an effect, any part of the game — into a self-contained task for an AI agent, build a verifier that objectively grades the agent's attempt, run an agent against it, and analyze how it did. The craft we are testing is the verifier — can you measure an agent's work on real game development precisely and honestly?

The game
RoboBlast: TPS Demo — Godot 4.6 (gdquest-demos)
Time
4–8 focused hours. Timebox it.
You need
Godot 4.6, a coding agent (Claude Code / Gemini CLI / Cursor …) wired to the Godot MCP, git
You submit
A public fork of the game — the ablated-task branch (what the agent sees), the verifier, agent-run branch(es), and a viewable HTML writeup

1.Context

We build evaluation environments that measure how well AI coding agents perform on real game-development work — not toy puzzles. A good environment hands an agent a realistic task, lets it work autonomously, and grades the result objectively and reproducibly. The hard, interesting part is almost never the task — it's the grader. This assignment asks you to build one, end to end, on a real game.

2.The game

You will work on RoboBlast: TPS Demo, a polished third-person shooter in Godot 4.6 — a real game (character controller, camera, navigation, enemies, weapons, physics, HUD), not a prototype. Clone it, open it in Godot, and play it before you cut anything.

  • Repo: github.com/gdquest-demos/godot-4-3d-third-person-controller
  • Fork it (public) and build on the current master — we lock the base to that, so you don't pick or record a base commit.

3.Your task

Produce a complete, self-contained task derived from this game, plus everything needed to grade an agent's attempt at it.

Tooling — use the Godot MCP. Equip your agents with the Godot MCP — a Model Context Protocol server that lets an agent drive the Godot editor/engine (open scenes, inspect nodes, run the project, read errors). It matters in two places: (1) the agent you use to build this assignment works far better with it; and (2) the rollout agent you grade must have it available. Giving the graded agent real engine access keeps the run robust and fair — we want to measure the true difficulty gap of your task, not penalize the agent for being unable to touch Godot.
Step 1

Pick a substantial feature

Choose one real element of the game with enough depth to be interesting and enough surface to grade. It does not have to be pure code — it can be any part of the game: gameplay behavior, an animation, a 3D model, a material / shader, a VFX / particle effect, UI / HUD, or audio. Prefer a whole element or behavior over a one-line tweak. Non-code / visual elements are often the harder things to grade deterministically — leaning in is welcome, but you still owe an honest, reproducible check for whatever you pick.

Step 2

Turn it into a task the agent must solve

Remove or disable that element so the game is now missing it — a clean, self-contained change on its own branch/commit of your fork (this can touch scripts, scenes, resources, animations, models, or UI, whatever the element is). This branch is the exact version the agent sees. Write a behavioral specification alongside it — what the feature should do or look like, in player-facing terms, for the agent to implement.

  • The spec must be behavioral only: describe what happens, never how. No file names, class names, method names, node paths, or signal names. The agent should have to understand the codebase, not follow a recipe.
  • The ablation should be complete: don't leave commented-out originals or obvious stubs that let the agent reverse-engineer the answer instead of reasoning about the behavior.
Step 3

Build a deterministic verifier

Write an automated, headless grader that runs the game's real systems and checks the actual behavioral outcome of the feature. It should not be a single pass/fail — grade out of 100, with each important sub-behavior worth a share of the points, so the final score reflects how completely the agent implemented the feature (partial credit, not all-or-nothing). This scored grader is the centerpiece of the assignment.

Step 4

Run agents against your task

Give a coding agent only your behavioral spec and the ablated project (not the original, not your verifier), let it work autonomously, and capture its output (the diff). Use Claude Code at minimum (other agents welcome), give the graded agent the Godot MCP (fairness — see tooling), and run each agent at least 3 times — agents are stochastic, so one run tells you little; multiple show the spread and pass rate.

Step 5

Grade, analyze, and present

Run your verifier on the attempts and report the scores. Then analyze: what did the agent get right, what did it get wrong, and — critically — is each failure a real defect in the agent's work or an artifact of your verifier? Inspect the agent's actual output to decide, and present it as the HTML report described in deliverables.

4.What makes a good verifier (the real test)

Anyone can write an assertion. A trustworthy grader has to satisfy all of these at once:

Discrimination. It must fail the ablated game and pass the original — if it can't tell those apart, it measures nothing.
Robust to reward hacking. It must reject near-miss cheats that look right but don't reproduce the behavior. Write a few "almost right" fake solutions and confirm your verifier catches every one.
No false negatives. It must not fail a correct solution that's merely structured differently. Grade the observable outcome, not one implementation; if you can't grade a dimension without assuming an implementation, don't grade it.
Calibrated difficulty — a hard requirement. The agent must not easily reach 100%: a strong agent should land on a partial score (clearly progressing, but missing something real), not a clean sweep and not zero. If your first runs all score ~100%, make the task harder before you submit.
Evaluation integrity. The agent must not be able to reach the answer while it works — no original source, git history, your verifier, or an online copy of the game. Note your fork also holds the original (on master) and your verifier, so make sure the agent's working copy can't reach them via other branches or history. If it can read the solution, the score is meaningless — tell us how you prevented this.

5.Deliverables

Submit a public GitHub fork of the game, forked from the current master (the locked base). In the fork:

  • The ablated task on its own branch — the exact version the rollout agent is given (feature removed/disabled), with the behavioral spec (the prompt) alongside it.
  • The verifier (runnable headless; include the exact command) — kept out of the agent's reach (see evaluation integrity).
  • Your anti-cheat probes — the fake near-miss solutions proving the verifier isn't foolable, and evidence they were all caught.
  • Agent runs: Claude Code at minimum (more welcome), ≥3 runs each, with the Godot MCP available — each run's result on its own branch is welcome; report the diff, score, and which agent/model/version + tools.
  • A viewable HTML writeup (~1 page, opens in a browser, no build step) with visuals — before/after clips or screenshots + the agent's attempts. Cover why you chose this feature, how you calibrated difficulty, how you prevented cheating, and your failure analysis (real agent bug vs. verifier artifact). Show, don't just tell.

6.How we evaluate you

DimensionWhat we look for
Verifier validityDiscriminates (ablated fails, original passes), is deterministic and reproducible, grades real outcomes. The single most important axis.
Anti-hack robustnessYou thought adversarially and proved the grader rejects near-miss cheats without false-negativing valid variants.
Difficulty calibrationThe task lands in the partial-credit zone for a strong agent, on purpose, and you can explain why.
AnalysisYou inspected the agent's real output and can separate genuine agent defects from grader artifacts.
CraftClean diff, clear spec, readable verifier, reproducible run, honest writeup.

A small task with a bullet-proof verifier beats an ambitious task with a leaky one — we value a correct, honest end-to-end slice over breadth.

7.Notes & hints

Use the visual dimension to make it hard. A powerful way to keep the agent from an easy 100% is to require the feature to be correct visually, not just logically. Agents very often get the underlying logic right while the on-screen result is wrong — an effect that never actually renders, an animation that's off, a model or material that looks incorrect. If your task demands that the result looks right, and your verifier actually checks the visual outcome (e.g. rendered frames / screenshots, pixel or region checks) rather than only internal state, you'll raise the bar and land in the partial-score zone we're looking for. Figuring out how to grade "does it look right" reproducibly is exactly the kind of craft we want to see.
